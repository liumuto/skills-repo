# skill-meta

本仓库所有 skill 的 **SKILL.md 写作元规范**。任何新增或修改 `.agents/skills/<x>/SKILL.md` 之前，必须先以 **5 节范式**（定时机 + 立目标 + 理规则 + 给示例 + 划边界）写全，并通过 **4 条红线**自查。

它是「写其它 skill 的 skill」——所有其它 SKILL.md 都以本文件为参考标准。

---

## 定时机（何时触发本 skill）

- **自动**：编辑或新增 `.agents/skills/<name>/SKILL.md` 时。
- **关键词触发**：用户提到「写个 skill / 加个 skill / 优化 skill / skill 规范 / skill 写法 / SKILL.md / authoring」。
- **不触发**：仅改周边文档、未改动任何 SKILL.md 时。

## 立目标

让每一个新增 / 修改的 SKILL.md 在合入前都满足：**5 节范式 + 4 条红线**。

杜绝「触发条件模糊 / 目标宽泛 / 缺反例 / 没有边界」导致 AI **误调或滥用** skill。一句话口径：**AI 在「编辑某个 SKILL.md」时，输出一份 5 节齐备、红线全过的文档草案。**

---

## 理规则 —— 5 节范式（强制 CHECKPOINT，缺一不可）

> **STOP**：5 节任一缺失，都会留下「幻觉可乘」的余地。缺则不合格 → 补全后再合入。

| 节 | 必备元素 | 反约范例（不合格表达） |
| --- | --- | --- |
| **定时机** | 至少一种机制化触发条件（文件 glob / 用户意图 / 状态量）；标明 **自动 / 手动**；列出**不触发**场景 | 「需要时」「合适时」「编写代码时」、无法机械判定、缺「不触发」列举 |
| **立目标** | 一句话 = 主语 + 动词 + **可衡量 / 可观察 / 可执行**的结果 | 「帮助用户……」「让代码更规范」等无主语、无量化口径 |
| **理规则** | 命令式 / 代码块 / 决策表；可机械执行 | 「按规范处理」「合理判断」「视情况而定」等模糊口径 |
| **给示例** | 至少 **1 正 1 反**。正例贴近真实调用场景，反例覆盖常见误用 | 全是正例没有反例，或全是禁止项没有正例 |
| **划边界** | 能力上限 + 兜底策略（输出「信息不足」/ 转人工 / 转其它 skill）+ 安全/隐私边界 + 与相邻 skill 的边界 | 缺兜底策略、未声明与其它 skill 的协作边界 |

### Frontmatter 规则

```yaml
---
name: <kebab-case，与目录名一致>
description: <1-2 句中文 + 一句英文 "Use when ..."；含触发关键词清单>
---
```

- **name**：与目录名**严格一致**，kebab-case。
- **description**：
  - **中文部分**：1-2 句讲清 skill 解决什么、何时触发。
  - **英文部分**：`Use when ...` 这一句，描述自动触发条件（让 AI agent loader 能机械匹配）。
  - **关键词触发清单**：用「触发关键词：」前缀列出口语化触发词。
  - **禁止**在 description 出现「帮助处理一些事情」「专门处理 X」之类无信息描述。

### 4 条红线（每个新 skill 必过）

新增 skill 前必须逐项自查：

1. **有证据**：来源是真实 bug / 教训，能命中触发场景或架构红线。
2. **可观测**：触发条件明确到文件 glob、操作类型或 diff 形态。
3. **可校验**：范例可被自检 / 审查 / 原则对齐逐项验证。
4. **可复现**：何时触发、指向哪些 SKILL.md、单跑可复现 → 文档化。

> 不达标前**不写入** skill 目录；先以草稿形式观察，补全后再合入。

## 给示例

### 示例 1：合格的 frontmatter

✅ **正例**：

```yaml
---
name: tracking-workflow
description: 埋点工作流分诊与失败排查链。Use when user mentions 埋点验收 / 埋点联调 / 落库 / 接入新埋点 / 退休埋点。触发关键词：埋点验收 / 埋点联调 / ……
---
```

✅ **通过**：触发关键词清单 + 英文 `Use when` + 中文目标描述齐全，AI agent loader 能机械匹配。

❌ **反例**：

```yaml
---
name: my-skill
description: 帮助处理一些 UI 相关的事情
---
```

❌ **失败原因**：目标宽泛、无触发条件、无关键词清单；AI 永远不知道何时调用。

### 示例 2：合格的「定时机」节

✅ **正例**：

```markdown
## 触发时机
- **自动**：编辑 `feature/**/*.kt` 或 `app/**/ui/**/*.kt` 中含 `@Composable` 的文件时
- **关键词触发**：用户提到 设置页 / 弹窗 / Dialog / 标题栏 / Switch / 卡片 / 列表项 等
- **新页面**：创建新 Screen / Dialog / Settings Row 时
- **不触发**：仅修改 string 资源命名、仅调整缩进格式
```

❌ **反例**：

```markdown
## 触发时机
编写 Compose UI 时使用本 skill。
```

❌ **失败原因**：「编写时」是循环定义（编写代码本来就是常态）；缺关键词清单、缺不触发场景，AI 不知何时停手。

### 示例 3：合格的「给示例」节

✅ **正例**（典型结构）：

- 给一个真实文件中的代码片段作为正例，逐句解释为何符合规则。
- 给一个常见误用作为反例，标注 ❌ 失败原因。
- 必要时给「何时不该用此 skill」表。

❌ **反例**：

- 只列了「禁止项」表，没说正确做法应该如何。
- 示例用伪代码（`foo()` / `bar()`），脱离项目真实场景。

---

## 划边界

- **只管文档结构**：本 skill 管「SKILL.md 写得合不合格」，不管 skill 内部的具体业务约束是否正确——后者由各 skill 自身负责。
- **不替代 AskUserQuestion**：当 SKILL.md 缺关键 input（证据来源、触发口径不明）时 → 先问明依据再写，不臆造。
- **与相邻 skill 边界**：新增 skill 与已有 skill 职责重叠时，先在本 skill 明确「不替代 X / 协同 Y」，再落地。
- **安全红线**：写入 SKILL.md 不得包含凭据、密钥、内网链接、敏感个人信息。

## 反例速查表（黑名单 8 条）

写完 SKILL.md 后**必查**；命中任一条 = 不合格，必须改：

| # | 反例 | 命中点 | 修正方向 |
| --- | --- | --- | --- |
| 1 | 循环定义触发条件 | 「编写代码时 / 需要时 / 合适时」 | 改为 glob pattern + 关键词清单 |
| 2 | frontmatter description 含「帮助处理一些事情」 | description 字段 | 删除空话；改为「做什么 + 何时用 + 触发词」 |
| 3 | 立目标无主语、无衡量口径 | 「让代码更规范 / 提升质量」 | 改为「AI 在 X 时输出 Y」 |
| 4 | 理规则用「按规范处理 / 视情况而定」 | 理规则段任何位置 | 改为命令 / 代码块 / 决策表 |
| 5 | 给示例全是正例（无反例） | 给示例段 | 至少 1 正 1 反，反例覆盖常见误用 |
| 6 | 给示例全是禁止项（无正例） | 给示例段 | 至少 1 正 1 反，正例贴近真实场景 |
| 7 | 划边界缺兜底策略 | 划边界段 | 加「信息不足 → AskUserQuestion / 转人工 / 转其它 skill」 |
| 8 | 未声明与相邻 skill 边界 | 划边界段 | 明列「不替代 X / 协同 Y」 |
